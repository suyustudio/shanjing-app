# 山径APP - 用户反馈收集方案

> **文档版本**: v1.0  
> **制定日期**: 2026-03-19  
> **文档状态**: 规划阶段  
> **优先级**: P2  
> **目标版本**: M4 后续迭代或 M5

---

## 1. 方案概述

### 1.1 设计目标

建立轻量、高效的用户反馈收集体系，支持内测及后续版本迭代。

| 目标维度 | 具体目标 | 衡量指标 |
|---------|---------|---------|
| **收集效率** | 降低用户反馈门槛 | 反馈提交步骤 < 3 步 |
| **信息完整** | 收集足够多的诊断信息 | 自动附加设备/版本信息 |
| **处理效率** | 快速响应用户反馈 | 24h 内首次响应 |
| **数据驱动** | 反馈数据指导产品决策 | 定期输出反馈分析报告 |

### 1.2 方案范围

| 范围 | 包含内容 |
|------|---------|
| **应用内入口** | 反馈按钮、快捷入口、场景化触发 |
| **反馈分类** | 问题类型、优先级、功能模块 |
| **内容收集** | 文字描述、截图、日志、联系方式 |
| **后台管理** | 反馈查看、分类、处理、统计 |

## 2. 应用内反馈入口设计

### 2.1 入口位置

| 入口位置 | 触发方式 | 适用场景 | 优先级 |
|---------|---------|---------|--------|
| **设置页** | 点击"帮助与反馈" | 用户主动反馈 | P0 |
| **导航页** | 悬浮按钮/菜单中 | 导航中遇到问题 | P1 |
| **错误页面** | 自动弹出反馈提示 | 崩溃/错误后 | P1 |
| **路线详情** | 下拉或更多菜单 | 路线数据问题 | P2 |
| **我的页** | 快捷入口 | 一般反馈 | P2 |

### 2.2 入口设计规范

**设置页入口（主入口）:**

```
设置页面
├── 账号与安全
├── 外观
│   └── 深色模式
├── 通知
├── 关于
│   ├── 关于山径
│   ├── 隐私政策
│   ├── 用户协议
│   └── 给我们评分
├── ─────────────────
└── ❓ 帮助与反馈 ← 主入口
    ├── 常见问题
    ├── 功能建议
    ├── 问题反馈 ← 进入反馈页
    └── 联系客服
```

**导航页快捷入口:**

```
导航页底部工具栏
├── [语音]
├── [放大]
├── [缩小]
├── [图层]
└── [更多] ← 展开菜单
    ├── 分享位置
    ├── 上报问题 ← 快捷反馈
    └── 结束导航
```

### 2.3 场景化触发

| 场景 | 触发时机 | 反馈类型 |
|------|---------|---------|
| 导航偏航 | 连续偏航3次 | 导航准确性反馈 |
| 定位失败 | GPS信号弱>30秒 | 定位问题反馈 |
| 路线完成 | 完成后评分<3星 | 路线体验反馈 |
| APP崩溃 | 重启后 | 崩溃问题反馈 |
| 离线地图失败 | 下载失败 | 离线功能反馈 |

## 3. 反馈分类体系

### 3.1 问题类型（一级分类）

| 类型ID | 类型名称 | 说明 | 示例 |
|--------|---------|------|------|
| `bug` | 功能异常 | 功能无法正常使用 | 点击无反应、闪退 |
| `crash` | 崩溃闪退 | APP意外关闭 | 导航中突然退出 |
| `ui` | 界面问题 | 显示异常 | 文字重叠、图片不显示 |
| `performance` | 性能问题 | 卡顿、慢 | 地图加载慢、卡顿 |
| `data` | 数据问题 | 路线/POI数据错误 | 坐标偏差、信息错误 |
| `navigation` | 导航问题 | 定位、偏航问题 | 定位不准、语音不播报 |
| `feature` | 功能建议 | 新功能建议 | 建议增加某功能 |
| `experience` | 体验优化 | 交互改进建议 | 按钮位置调整 |
| `other` | 其他 | 不属于以上类别 | |

### 3.2 功能模块（二级分类）

| 模块ID | 模块名称 | 关联问题类型 |
|--------|---------|-------------|
| `home` | 首页/发现 | bug, ui, performance |
| `search` | 搜索 | bug, data, performance |
| `detail` | 路线详情 | bug, ui, data |
| `navigation` | 导航 | bug, crash, navigation |
| `offline` | 离线地图 | bug, performance |
| `user` | 用户中心 | bug, ui |
| `share` | 分享功能 | bug, ui |
| `sos` | SOS功能 | bug, crash |
| `auth` | 登录注册 | bug, ui |
| `other` | 其他 | other |

### 3.3 严重程度

| 级别 | 名称 | 说明 | 响应时效 |
|------|------|------|---------|
| `critical` | 严重 | 核心功能不可用，影响安全 | 2h |
| `high` | 高 | 重要功能异常，体验差 | 24h |
| `medium` | 中 | 一般问题，有workaround | 72h |
| `low` | 低 | 轻微问题，建议类 | 7d |

### 4.2 表单字段规范

| 字段 | 类型 | 必填 | 验证规则 | 说明 |
|------|------|------|---------|------|
| `type` | 单选 | 是 | 选择一级分类 | 默认"功能异常" |
| `module` | 单选 | 是 | 选择二级分类 | 根据type联动 |
| `severity` | 单选 | 否 | 四选一 | 默认"高" |
| `description` | 文本 | 是 | 10-500字 | 富文本输入 |
| `images` | 图片 | 否 | 最多3张，每张<5MB | 支持拍照/相册 |
| `contact` | 文本 | 否 | 手机或微信 | 用于回访 |
| `attach_log` | 开关 | 否 | 布尔值 | 默认开启 |

## 5. 反馈处理流程

### 5.1 处理流程图

```
用户提交反馈
    ↓
[系统自动] 分类、打标签、去重
    ↓
[系统自动] 严重/crash 立即告警
    ↓
人工初审（24h内）
    ↓
    ├─→ 重复问题 → 关联已有工单
    ├─→ 已修复 → 回复用户
    ├─→ 需要更多信息 → 联系用户
    └─→ 确认问题 → 创建Bug工单
                ↓
            分配开发人员
                ↓
            修复 → 测试 → 发布
                ↓
            回复用户已解决
```

### 5.2 SLA 定义

| 严重程度 | 首次响应 | 处理完成 | 通知方式 |
|---------|---------|---------|---------|
| critical | 2小时 | 24小时 | 飞书告警+电话 |
| high | 24小时 | 72小时 | 飞书通知 |
| medium | 72小时 | 7天 | 邮件通知 |
| low | 7天 | 排期处理 | 系统内通知 |

### 5.3 反馈标签体系

| 标签类型 | 标签示例 |
|---------|---------|
| 状态 | `待处理`, `处理中`, `待验证`, `已解决`, `已关闭` |
| 优先级 | `P0`, `P1`, `P2`, `P3` |
| 版本 | `v1.0.0`, `v1.1.0` |
| 来源 | `内测`, `公测`, `生产` |
| 设备 | `Android`, `iOS` |
| 特殊 | `需回访`, `需日志`, `需复现` |

## 7. 技术实现建议

### 7.1 数据库设计

```sql
-- 反馈表
CREATE TABLE feedbacks (
    id SERIAL PRIMARY KEY,
    user_id VARCHAR(64),
    type VARCHAR(32) NOT NULL,
    module VARCHAR(32) NOT NULL,
    severity VARCHAR(16) DEFAULT 'medium',
    description TEXT NOT NULL,
    images JSON,
    contact VARCHAR(64),
    attach_log BOOLEAN DEFAULT true,
    device_info JSON,
    context JSON,
    status VARCHAR(32) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    resolved_at TIMESTAMP
);

-- 反馈处理记录
CREATE TABLE feedback_actions (
    id SERIAL PRIMARY KEY,
    feedback_id INTEGER REFERENCES feedbacks(id),
    action_type VARCHAR(32), -- reply, status_change, assign
    action_by VARCHAR(64),
    action_content TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 7.2 API 设计

```
POST /api/feedback/submit
提交反馈

GET /api/feedback/list
获取反馈列表（管理后台）

GET /api/feedback/:id
获取反馈详情

POST /api/feedback/:id/reply
回复反馈

POST /api/feedback/:id/status
更新反馈状态
```

### 7.3 实现优先级

| 阶段 | 功能 | 工时估算 |
|------|------|---------|
| Phase 1 | 基础反馈提交 + 列表查看 | 12h |
| Phase 2 | 自动分类 + 数据统计 | 8h |
| Phase 3 | 高级筛选 + 知识库 | 8h |

## 8. 内测期简化方案

考虑到 M4 内测阶段的时间限制，建议先实施**简化版**：

### 8.1 简化功能

| 功能 | 简化方案 |
|------|---------|
| 入口 | 仅设置页入口 |
| 分类 | 仅问题类型（单选），不做二级分类 |
| 字段 | 类型 + 描述 + 截图（3项） |
| 自动信息 | 仅设备信息和用户ID |
| 后台 | 简单的列表查看，飞书通知 |

### 8.2 简化后工时

- **客户端**: 6h（表单页面 + API调用）
- **后端**: 6h（API + 数据库 + 飞书通知）
- **总计**: 12h
