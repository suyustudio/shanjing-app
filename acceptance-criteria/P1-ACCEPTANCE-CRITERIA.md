# M7 P1 收藏夹增强功能 - P1问题验收标准

**文档版本**: 1.0  
**制定日期**: 2026-03-22  
**制定人员**: Product Agent  
**适用范围**: 方案B第一阶段（3月23日-3月26日）P1问题修复验收

## 概述

本文档定义了M7 P1收藏夹增强功能中6个P1级别问题的验收标准。这些标准用于验证修复是否满足产品要求，确保功能可测试、可验证。

## 一、P1-01：批量删除使用逐个API问题（性能问题）

### 问题描述
批量删除操作使用循环逐个调用`removeTrailFromCollection` API，而非批量删除API，导致性能低下，网络请求增多。

### 验收标准

#### 1. 功能验收标准
1. **批量API调用验证**:
   - 当用户选择多个路线并执行批量删除时，系统必须调用`batchRemoveTrailsFromCollection`批量API
   - 网络监控显示只有一个API请求发送，包含所有选中的路线ID列表
   - 请求体格式正确：`{"trailIds": ["id1", "id2", "id3"], "collectionId": "xxx"}`

2. **操作结果一致性**:
   - 批量删除成功后，所有选中的路线从收藏夹中移除
   - UI立即更新，移除对应的路线卡片
   - 选中数量计数器归零，批量操作栏自动隐藏

3. **数据一致性**:
   - 删除后收藏夹中的路线总数减少对应的数量
   - 搜索过滤结果同步更新
   - 标签关联关系自动清理

#### 2. 性能验收标准
1. **响应时间**:
   - 批量删除10个路线的操作总时间 ≤ 3秒（从点击到UI更新完成）
   - 网络请求延迟 ≤ 2秒
   - UI响应时间 ≤ 1秒

2. **网络效率**:
   - 删除N个路线时，网络请求数量为1（批量API）
   - 请求数据量不超过`100 + N*50`字节
   - 无重复请求或冗余请求

3. **资源使用**:
   - 内存使用在批量操作期间增长不超过50MB
   - CPU占用峰值不超过80%
   - 无内存泄漏（操作完成后内存回落到初始水平）

#### 3. 用户体验验收标准
1. **进度反馈**:
   - 批量删除执行期间显示进度指示器（如旋转加载图标）
   - 进度文字提示："正在删除X个路线..."
   - 操作完成后显示成功提示："成功删除X个路线"

2. **错误处理**:
   - 部分失败时显示详细错误信息："成功删除Y个路线，Z个路线删除失败"
   - 失败时提供重试选项
   - 网络异常时显示友好错误提示

3. **撤销能力**:
   - 删除操作完成后，在5秒内显示"撤销"选项
   - 点击撤销可恢复删除的路线（需API支持）

#### 4. 可测试验证方法
1. **单元测试**:
   - `CollectionEnhancedService.batchRemoveTrailsFromCollection`方法测试覆盖率达到100%
   - 模拟批量删除成功/失败场景测试
   - 边界测试：删除0个、1个、最大数量限制的路由

2. **集成测试**:
   - 端到端测试：选择多个路线 → 点击批量删除 → 验证API调用和UI更新
   - 网络模拟测试：模拟慢速网络、网络中断场景
   - 并发测试：同时执行多个批量操作

3. **性能测试**:
   - 使用性能分析工具测量批量删除的响应时间
   - 网络抓包验证请求数量和数据量
   - 内存泄漏检测：重复执行批量删除操作100次

4. **手动测试用例**:
   - **TC-P1-01-01**: 选择2个路线，执行批量删除，验证批量API调用
   - **TC-P1-01-02**: 选择10个路线，执行批量删除，验证响应时间<3秒
   - **TC-P1-01-03**: 网络断开时执行批量删除，验证错误处理
   - **TC-P1-01-04**: 删除成功后验证UI更新和计数归零

## 二、P1-02：全选逻辑错误（用户体验问题）

### 问题描述
搜索过滤后，全选按钮会选中所有路线（包括未显示的），而非仅选中当前显示的过滤后路线，违背用户直觉。

### 验收标准

#### 1. 功能验收标准
1. **搜索过滤后全选行为**:
   - 当搜索结果有过滤时，点击"全选"按钮仅选中当前显示的所有路线
   - 选中数量显示为"X/Y选中"，其中X为当前显示路线数，Y为总路线数
   - 未显示的路线不应被选中

2. **全选状态同步**:
   - 全选按钮状态与当前显示路线的选中状态同步
   - 如果当前显示的所有路线都已选中，全选按钮显示为"全选取消"
   - 如果部分选中，全选按钮显示为"全选当前结果"

3. **搜索状态切换**:
   - 清空搜索框后，全选状态自动重置
   - 搜索关键词变化时，选中状态保持不变但针对新的过滤结果
   - 退出多选模式后重新进入，选中状态重置

#### 2. 用户体验验收标准
1. **视觉反馈**:
   - 搜索过滤状态下，全选按钮有视觉区别（如添加搜索图标或不同颜色）
   - 选中数量显示明确区分"当前结果/总数量"
   - 提供提示文字："全选当前搜索结果（X项）"

2. **交互一致性**:
   - 用户可直观理解全选的范围限制
   - 操作结果符合用户预期（仅操作看到的项目）
   - 提供状态提示，防止误操作

3. **可访问性**:
   - 屏幕阅读器正确读取全选状态和范围
   - 键盘导航支持全选操作
   - 高对比度模式下全选状态清晰可见

#### 3. 数据一致性验收标准
1. **选中状态存储**:
   - 选中状态正确存储在`CollectionSelectionManager`中
   - 搜索过滤变化时，选中状态映射到正确的路线ID
   - 批量操作使用正确的选中路线ID列表

2. **批量操作范围**:
   - 批量删除、移动、标签操作仅作用于当前选中的路线
   - 操作结果与用户看到的选中范围一致
   - 操作完成后，选中状态正确清除

#### 4. 可测试验证方法
1. **单元测试**:
   - `CollectionSelectionManager`搜索过滤逻辑测试
   - 全选按钮状态计算逻辑测试
   - 边界测试：空搜索结果、单个结果、大量结果

2. **集成测试**:
   - 搜索后全选 → 执行批量操作 → 验证操作范围
   - 搜索关键词变化对选中状态的影响测试
   - 多选模式进入/退出状态重置测试

3. **手动测试用例**:
   - **TC-P1-02-01**: 搜索过滤后点击全选，验证仅选中显示路线
   - **TC-P1-02-02**: 清空搜索后验证全选状态重置
   - **TC-P1-02-03**: 部分选中时全选按钮状态验证
   - **TC-P1-02-04**: 全选后执行批量删除，验证仅删除显示路线

## 三、P1-03：批量标签功能缺失（功能完整性问题）

### 问题描述
批量标签操作仅显示"此功能即将上线"占位对话框，实际功能未实现。

### 验收标准

#### 1. 功能验收标准
1. **批量添加标签功能**:
   - 选择多个路线后，点击"标签"按钮打开标签选择器
   - 支持选择现有标签或创建新标签
   - 确认后为所有选中路线添加指定标签
   - 标签添加成功后在路线卡片上立即显示

2. **批量移除标签功能**:
   - 选择多个路线后，长按标签按钮或通过二级菜单选择"移除标签"
   - 显示选中路线共有的标签列表
   - 选择要移除的标签，确认后从所有选中路线移除
   - 标签移除成功后UI立即更新

3. **标签选择器功能**:
   - 显示所有可用标签列表，带颜色编码
   - 支持搜索过滤标签
   - 支持创建新标签（输入名称，自动分配颜色）
   - 多选支持：可同时选择多个标签批量添加

#### 2. 用户体验验收标准
1. **操作流程**:
   - 批量标签操作流程直观：选择路线 → 点击标签 → 选择标签 → 确认
   - 提供明确的进度反馈："正在为X个路线添加标签..."
   - 操作成功提示："成功为X个路线添加标签Y"

2. **视觉反馈**:
   - 路线卡片上的标签立即更新显示
   - 批量操作期间有加载状态指示
   - 错误情况有明确提示

3. **容错处理**:
   - 部分路线标签添加失败时提供详细错误信息
   - 支持重试失败的操作
   - 网络异常时友好提示

#### 3. 数据一致性验收标准
1. **API集成**:
   - 使用`batchAddTagsToTrails`批量API添加标签
   - 使用`batchRemoveTagsFromTrails`批量API移除标签
   - 请求格式正确：`{"trailIds": [...], "tagIds": [...]}`

2. **状态同步**:
   - 标签添加/移除后，本地状态立即更新
   - 与服务器数据保持同步
   - 离线状态下操作排队，联网后同步

#### 4. 性能验收标准
1. **响应时间**:
   - 打开标签选择器时间 ≤ 500ms
   - 批量添加标签操作总时间 ≤ 3秒（10个路线）
   - UI更新响应时间 ≤ 1秒

2. **API调用**:
   - 批量标签操作使用单个API请求
   - 无重复或冗余请求

#### 5. 可测试验证方法
1. **单元测试**:
   - `TagService.batchAddTagsToTrails`和`batchRemoveTagsFromTrails`测试
   - 标签选择器组件测试
   - 批量标签操作控制器测试

2. **集成测试**:
   - 端到端测试：选择路线 → 批量添加标签 → 验证UI更新
   - 批量移除标签流程测试
   - 网络异常场景测试

3. **手动测试用例**:
   - **TC-P1-03-01**: 选择多个路线，批量添加新标签，验证所有路线显示该标签
   - **TC-P1-03-02**: 批量移除标签，验证标签从所有选中路线移除
   - **TC-P1-03-03**: 标签选择器搜索和创建新标签测试
   - **TC-P1-03-04**: 部分成功场景测试，验证错误处理

## 四、P1-04：防抖延迟不符合规格（优化问题）

### 问题描述
搜索防抖延迟为500ms，不符合规格要求的300ms，导致搜索响应速度低于预期。

### 验收标准

#### 1. 功能验收标准
1. **防抖时间调整**:
   - 搜索输入防抖时间从500ms调整为300ms
   - 输入停止300ms后自动触发搜索
   - 连续输入时，防抖计时器正确重置

2. **搜索响应**:
   - 搜索请求在防抖时间结束后立即发送
   - 搜索结果在请求返回后立即显示
   - 搜索过程中显示加载状态

3. **取消机制**:
   - 输入新字符时取消之前的待处理搜索
   - 清空搜索框时立即取消搜索并显示原始列表

#### 2. 性能验收标准
1. **响应时间**:
   - 从停止输入到显示搜索结果的总时间 ≤ 600ms
   - 防抖延迟：300ms ± 10%
   - 搜索结果渲染时间 ≤ 200ms

2. **用户体验指标**:
   - 快速输入时无明显延迟感
   - 搜索结果更新流畅，无卡顿
   - 内存使用稳定，无泄漏

#### 3. 用户体验验收标准
1. **即时反馈**:
   - 输入时显示"正在输入..."或类似提示
   - 搜索执行时显示加载动画
   - 空搜索结果时显示空状态提示

2. **可感知性能**:
   - 用户感知搜索响应迅速
   - 无明显的输入到结果显示的延迟
   - 流畅的搜索体验

#### 4. 可测试验证方法
1. **单元测试**:
   - `SearchController`防抖逻辑测试，验证300ms延迟
   - 防抖计时器重置逻辑测试
   - 搜索取消机制测试

2. **性能测试**:
   - 使用性能分析工具测量实际防抖时间
   - 搜索响应时间测量
   - 内存使用监控

3. **手动测试用例**:
   - **TC-P1-04-01**: 快速输入多个字符，验证300ms后触发搜索
   - **TC-P1-04-02**: 测量从停止输入到显示结果的时间，验证≤600ms
   - **TC-P1-04-03**: 连续输入测试，验证防抖计时器重置
   - **TC-P1-04-04**: 清空搜索框测试，验证立即取消搜索

## 五、P1-05：网络错误处理不完善（可靠性问题）

### 问题描述
网络异常时，批量操作缺少错误处理，可能导致静默失败，用户无法感知问题。

### 验收标准

#### 1. 功能验收标准
1. **网络异常检测**:
   - 检测到网络连接失败时立即提示用户
   - 区分不同类型的网络错误（无网络、超时、服务器错误等）
   - 提供具体的错误信息，而非通用提示

2. **批量操作错误处理**:
   - 批量操作部分失败时显示详细报告
   - 提供"重试失败项"选项
   - 失败的路由保持选中状态以便重试

3. **离线模式支持**:
   - 网络断开时提示用户处于离线状态
   - 允许排队操作，联网后自动执行
   - 离线时可查看本地数据，但禁用需要网络的操作

#### 2. 用户体验验收标准
1. **错误提示友好性**:
   - 错误消息使用用户易懂的语言，避免技术术语
   - 提供解决问题建议（如检查网络、稍后重试）
   - 错误提示持续时间合理（3-5秒）

2. **恢复机制**:
   - 操作失败后提供明确的恢复路径
   - 支持部分重试（仅重试失败项）
   - 操作历史记录，便于追溯问题

3. **状态可见性**:
   - 网络状态在UI中可见（如状态栏图标）
   - 批量操作进度和状态清晰可见
   - 错误状态有明确的视觉标识

#### 3. 可靠性验收标准
1. **错误恢复率**:
   - 网络恢复后，排队操作自动重试成功率 ≥ 95%
   - 用户手动重试操作成功率 ≥ 90%
   - 部分失败场景下，成功项持久化率100%

2. **数据完整性**:
   - 操作失败时无数据丢失或损坏
   - 部分成功操作的数据保持一致性
   - 重试操作不会导致数据重复或冲突

#### 4. 可测试验证方法
1. **单元测试**:
   - 网络错误处理逻辑测试
   - 批量操作错误恢复测试
   - 离线模式队列管理测试

2. **集成测试**:
   - 模拟网络断开场景测试
   - 批量操作部分失败场景测试
   - 网络恢复自动重试测试

3. **手动测试用例**:
   - **TC-P1-05-01**: 断开网络执行批量删除，验证错误提示
   - **TC-P1-05-02**: 网络超时测试，验证重试机制
   - **TC-P1-05-03**: 批量操作部分失败测试，验证详细报告
   - **TC-P1-05-04**: 网络恢复后验证排队操作自动执行

## 六、P1-06：权限检查未实现（安全问题）

### 问题描述
未检查用户对收藏夹的操作权限，可能导致越权操作，存在安全风险。

### 验收标准

#### 1. 功能验收标准
1. **权限验证机制**:
   - 所有收藏夹操作前验证用户权限
   - 权限检查包括：查看、编辑、删除、分享权限
   - 权限不足时阻止操作并提示用户

2. **前端权限检查**:
   - 加载收藏夹时验证查看权限
   - 进入编辑/多选模式时验证编辑权限
   - 执行删除/移动操作时验证删除权限

3. **后端权限验证**:
   - 所有API端点包含权限验证
   - 返回明确的权限错误代码（403 Forbidden）
   - 日志记录权限验证失败事件

#### 2. 安全验收标准
1. **越权防护**:
   - 用户无法访问无权限的收藏夹
   - 用户无法修改无编辑权限的收藏夹
   - 用户无法删除无删除权限的收藏夹

2. **数据隔离**:
   - 用户数据严格隔离，无法访问他人数据
   - 权限变更后立即生效
   - 缓存数据包含权限标记，过期后重新验证

3. **安全审计**:
   - 所有权限检查操作记录日志
   - 异常权限尝试触发安全警报
   - 定期审计权限配置和使用

#### 3. 用户体验验收标准
1. **权限提示友好性**:
   - 权限不足时显示友好提示，而非技术错误
   - 提供获取权限的途径（如联系管理员、升级账户）
   - 提示内容符合用户角色和理解能力

2. **界面适配**:
   - 无权限的操作按钮置灰或隐藏
   - 权限变化时界面实时更新
   - 多用户协作场景权限显示清晰

#### 4. 性能验收标准
1. **权限检查性能**:
   - 权限检查增加的总延迟 ≤ 100ms
   - 权限缓存有效减少重复检查
   - 批量操作权限检查优化为单次检查

#### 5. 可测试验证方法
1. **单元测试**:
   - 权限验证逻辑测试
