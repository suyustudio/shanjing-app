## Safety Feature PRD v1.0

**文档版本**: 1.0  
**创建日期**: 2026-03-19  
**作者**: Product Agent  
**状态**: 草案

## 1. 文档概述

### 1.1 背景
当前山径APP已实现基础安全中心页面，但用户反馈SOS按钮设计存在问题（已移至"我的"页面）。需要重新设计整个安全功能体系，参考AllTrails Lifeline的设计理念，构建更符合国内用户习惯的安全保障系统。

### 1.2 目标
- 重新定义安全功能架构，区分"主动求救"与"被动监控"两类场景
- 设计完整的Lifeline功能，实现出发前设置、实时分享、超时报警的全流程
- 优化SOS功能，平衡易用性与防误触
- 保持积极、轻松的产品氛围，避免过度强调危险

### 1.3 参考竞品
- **AllTrails Lifeline**: 出发前设置紧急联系人和预计完成时间，实时位置分享，超时未归自动报警
- **Garmin inReach**: 独立物理SOS按钮+保护盖，专业卫星救援协调中心
- **两步路**: 留守救援、紧急呼救、位置分享

## 2. 安全功能架构设计

### 2.1 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                    山径APP 安全体系                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐      ┌──────────────┐                    │
│  │  主动求救     │      │  被动监控     │                    │
│  │  (用户触发)   │      │  (自动运行)   │                    │
│  └──────┬───────┘      └──────┬───────┘                    │
│         │                      │                            │
│         ▼                      ▼                            │
│  ┌──────────────┐      ┌──────────────┐                    │
│  │ 紧急SOS      │      │ Lifeline     │                    │
│  │ • 多层确认   │      │ • 行程分享   │                    │
│  │ • 位置发送   │      │ • 超时预警   │                    │
│  │ • 救援信息   │      │ • 自动报平安  │                    │
│  └──────────────┘      └──────────────┘                    │
│                                                             │
│  设计理念: 让安全成为习惯，而非恐惧的来源                      │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 安全功能双轨制

| 维度 | 主动求救 (SOS) | 被动监控 (Lifeline) |
|------|---------------|---------------------|
| **触发方式** | 用户主动触发 | 系统自动监控 |
| **使用场景** | 真正的紧急情况 | 日常徒步保障 |
| **用户干预** | 需要用户操作 | 设置后无需干预 |
| **心理负担** | 高（仅在危急时使用） | 低（常态化的安心保障） |
| **技术依赖** | 手机信号/卫星 | GPS+定时任务 |
| **适合人群** | 所有用户 | 独行用户、长距离用户 |

### 2.3 设计原则

1. **安全第一**: 任何设计决策优先考虑用户生命安全
2. **防误触**: 紧急功能必须有可靠的防误触机制
3. **无负担**: 被动监控功能不应增加用户心理负担
4. **积极氛围**: 使用正面、鼓励性的文案，避免恐吓
5. **离线优先**: 核心安全功能在无网络环境下仍能工作
6. **合规合法**: 符合国内隐私保护和数据安全法规

## 3. Lifeline 功能设计

### 3.1 功能概述

Lifeline是山径APP的**被动安全监控系统**，用户在出发前设置紧急联系人和预计完成时间，系统会在后台自动运行，无需用户持续操作。

### 3.3 功能模块详解

#### 3.3.1 出发设置流程

**触发时机**: 用户在路线详情页点击"开始徒步"时

**设置步骤**:
1. **紧急联系人选择** (最多5人)
   - 从通讯录导入
   - 从历史联系人选择
   - 手动输入（姓名+手机号）
   - 标记"主要联系人"(1人，优先接收通知)

2. **预计完成时间设置**
   - 基于路线数据智能推荐（参考路线平均用时+用户历史速度）
   - 用户可手动调整
   - 缓冲时间设置（默认30分钟，可选15/30/60分钟）

3. **分享方式**
   - 微信分享（默认）
   - 短信分享
   - 复制链接

**UI文案示例**:
```
标题: 开启行程守护
副标题: 让关心你的人知道你很安全

联系人: 选择1-5位紧急联系人
预计用时: 系统推荐 3小时30分钟 [调整]
缓冲时间: 额外预留 30分钟 [调整]

[一键开启行程守护]
```

#### 3.3.2 实时位置分享机制

**技术方案**:
- 使用GPS记录轨迹点
- 有网络时：实时上传位置，联系人可查看实时地图
- 无网络时：本地缓存，恢复网络后批量上传
- 频率：每5分钟或每移动500米（取较频繁者）

**联系人视角**:
- 收到微信小程序卡片/短信链接
- 点击后打开地图页面，显示：
  - 用户当前位置（实时或最近已知）
  - 规划路线
  - 已行进轨迹
  - 预计完成时间
  - "用户最后更新：5分钟前"

#### 3.3.4 信号弱/无信号处理策略

| 场景 | 处理策略 |
|------|----------|
| 间歇性信号 | 本地缓存位置点，恢复信号后批量上传 |
| 长时间无信号 | 继续记录GPS轨迹，保存到本地 |
| 超时报警时无信号 | 发送"最后已知位置"+"信号丢失时间" |
| 恢复信号后 | 自动同步离线期间的所有位置点 |

**关键设计**: 即使完全无信号，Lifeline仍在本地记录轨迹，确保救援时能提供完整路线信息。

## 4. SOS功能优化设计

### 4.1 设计目标

- 在真正的紧急情况下快速触发
- 有效防止误触
- 与系统紧急呼叫形成互补而非替代
- 提供足够信息帮助救援

### 4.2 触发条件设计

**方案对比**:

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| A | 长按3秒+确认弹窗 | 简单直接 | 紧急时可能增加操作负担 |
| B | 快速连按5次电源键 | 与iPhone/Android SOS一致 | 需用户学习，可能误触 |
| C | 长按5秒（无确认） | 最快触发 | 误触风险高 |
| D | **长按5秒+滑动确认** ⭐ | 平衡速度与防误触 | 需要一定学习成本 |

**推荐方案D**: 长按5秒+滑动确认
- 长按触发振动反馈，进入紧急模式界面
- 显示10秒倒计时（可取消）
- 倒计时结束后自动发送
- 或用户主动滑动"立即发送"

### 4.4 发送内容

**SOS消息模板**:

```
【山径APP紧急求救】

姓名：{userName}
电话：{userPhone}

紧急状况！我需要帮助！

最后位置：{latitude}, {longitude}
位置精度：{accuracy}米
海拔：{altitude}米
更新时间：{timestamp}

当前路线：{routeName}
预计位置：{routeLocation}

紧急备注：{emergencyNote}

点击查看地图：{mapUrl}

---
此消息由山径APP自动发送
```

### 4.5 与系统紧急呼叫的关系

| 功能 | 山径APP SOS | 系统紧急呼叫(110/120) |
|------|-------------|----------------------|
| **触发方式** | APP内操作 | 拨号/快捷键 |
| **信息传递** | 发送位置+路线信息给联系人 | 语音通话 |
| **响应方** | 紧急联系人 | 专业救援机构 |
| **优势** | 自动发送精确位置 | 直接对接专业救援 |
| **使用建议** | 先通知联系人 | 严重紧急时直接拨打 |

**产品引导**:
```
重要提示：
山径SOS会通知您设置的紧急联系人。
如遇生命危险，请直接拨打 110/120。
```

### 4.6 防误触机制

1. **物理位置**: SOS按钮放置在二级菜单（"我的"→"安全中心"），不在地图首页
2. **操作复杂度**: 长按5秒+滑动确认，避免误触
3. **取消机会**: 10秒倒计时期间可随时取消
4. **重复触发限制**: 触发后30分钟内再次触发需输入密码确认
5. **确认机制**: 首次发送后，如用户无响应，会再次提示确认

## 6. 功能优先级

### 6.1 优先级矩阵

| 优先级 | 功能 | 用户价值 | 技术难度 | 备注 |
|--------|------|----------|----------|------|
| **P0** | Lifeline 核心流程 | 高 | 中 | MVP必选 |

## 7. 与竞品的差异化设计

### 7.1 竞品对比分析

| 功能 | AllTrails Lifeline | 两步路 | Garmin inReach | **山径APP** |
|------|-------------------|--------|----------------|------------|
| **SOS位置** | 无 | 地图右上角 | 物理按钮 | 二级菜单 |
| **联系人数量** | 1人 | 多人 | 多人 | **5人** |
| **超时报警** | ✅ | ✅ | ✅ | ✅ |
| **离线支持** | 有限 | 部分 | 卫星 | **优先设计** |
| **社交分享** | 弱 | 强 | 无 | **中等** |
| **国内救援** | 国外 | 国内 | 全球 | **专注国内** |
| **产品调性** | 工具化 | 社区化 | 专业级 | **轻量化** |

### 7.2 差异化亮点

#### 1. "安心模式"而非"恐惧营销"
- **竞品**: 强调危险、强调"万一"
- **山径**: 强调安心、强调"让关心你的人放心"
- **文案示例**: "让爱你的人随时知道你很安全" vs "防止你在山里失联死亡"

#### 2. 更适合中国用户的分享方式
- **竞品**: 依赖邮件/短信
- **山径**: 深度集成微信生态，小程序查看位置

#### 3. 离线优先设计
- **竞品**: 假设有网络
- **山径**: 核心功能在无网络下仍可用，恢复后自动同步

#### 4. 与路线深度整合
- **竞品**: 独立的安全功能
- **山径**: 安全功能与路线规划无缝融合，一键开启

#### 5. 适度的防误触
- **竞品**: 过于简单（易误触）或过于复杂（难触发）
- **山径**: 平衡设计，紧急情况可快速触发，日常使用不误触

### 8.2 关键技术点评估

| 技术点 | 可行性 | 风险 | 解决方案 |
|--------|--------|------|----------|
| 后台位置追踪 | ✅ 高 | 电量消耗 | 智能频率调整，静止时降频 |
| 离线数据缓存 | ✅ 高 | 存储限制 | 只保留关键位置点，定期清理 |
| 超时检测 | ✅ 高 | 时钟漂移 | 服务器时间为准 |
| 短信发送 | ✅ 高 | 成本 | 与短信服务商合作 |
| 微信小程序 | ✅ 中 | 审核 | 提前准备审核材料 |
| 紧急呼叫对接 | ⚠️ 中 | 法规限制 | 先通知联系人，不直接对接110 |

### 8.3 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 位置精度 | ≤50米 | 户外开阔地带 |
| 位置上传延迟 | ≤30秒 | 有网络情况下 |
| SOS发送延迟 | ≤10秒 | 从触发到发送 |
| 离线存储时长 | ≥72小时 | 持续徒步场景 |
| 电量消耗 | ≤10%/小时 | 后台追踪模式 |

### 8.4 安全与隐私

**数据安全措施**:
- 位置数据端到端加密
- 分享链接带时效token（默认24小时）
- 用户可随时撤销分享
- 紧急联系人只能查看，不能下载历史轨迹

**合规要求**:
- 符合《个人信息保护法》
- 位置收集需明示同意
- 提供数据删除功能
- 未成年人需监护人同意

## 11. 风险评估与应对

### 11.1 功能风险

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| SOS误触导致骚扰 | 中 | 中 | 多层确认，30分钟冷却期 |
| 位置数据不准确 | 高 | 低 | GPS+网络混合定位，精度标注 |
| 超时报警不准确 | 中 | 中 | 用户可延长，误报可撤销 |
| 用户隐私担忧 | 中 | 中 | 明确告知数据使用，提供删除 |
| 电池消耗过快 | 中 | 中 | 智能频率调整，省电模式 |

### 11.2 法律合规风险

| 风险 | 应对措施 |
|------|----------|
| 位置数据收集合规 | 明示同意，最小必要原则 |
| 紧急联系人信息保护 | 加密存储，仅用于紧急通知 |
| 虚假求救信息 | 用户协议明确法律责任 |
| 救援失败的责任边界 | 免责声明，明确APP仅辅助 |
