# 构建失败问题分析报告

## 问题概述
从 2026-03-02 到 2026-03-03，共进行了 **40+ 次构建尝试**，绝大多数失败。

## 失败模式分析

### 1. 重复犯的错误

| 错误 | 次数 | 原因 |
|------|------|------|
| `android/settings.gradle not found` | 10+ | 每次都在修复同样的问题 |
| `flutter create` 目录结构不对 | 8+ | 没有理解 Flutter 3.19 的新行为 |
| 工作流文件冲突 | 5+ | 创建了多个工作流，互相干扰 |
| APK 路径错误 | 3+ | 没有确认实际构建输出路径 |
| 权限/签名配置 | 2+ | 硬编码密码，未使用 secrets |

### 2. 根本原因

#### A. 对 Flutter 3.19 的 Android 项目结构理解错误
- **错误假设**：`flutter create` 会创建 `android/settings.gradle`
- **实际情况**：Flutter 3.19 使用新的项目结构，`settings.gradle` 需要手动创建或在特定条件下生成
- **重复犯错**：每次构建失败都尝试用 `flutter create` 修复，而不是手动创建必要文件

#### B. 工作流管理混乱
- 创建了 `build.yml`, `build-fixed.yml`, `build-fixed-v2.yml`, `build-simple.yml`, `release-build.yml`
- 没有统一的管理，旧工作流持续触发，新工作流不生效
- 禁用/删除操作没有彻底执行

#### C. 缺乏验证步骤
- 每次修改后没有本地验证
- 没有检查 GitHub Actions 实际使用的文件
- 依赖模拟数据和猜测，而不是实际日志

#### D. 修复策略不当
- 每次失败后添加更多复杂性（更多的检查、更多的文件创建）
- 没有简化问题，反而让工作流更难维护
- 没有回滚到已知可用的状态

### 3. 成功的构建

**Build #33** (build-fixed.yml) - 唯一成功的构建
- 时间：2026-03-02 18:22
- 关键差异：使用了当时存在的 `android/` 目录结构
- 之后失败的原因：代码更新破坏了 Android 目录结构

### 4. 应该采取的正确策略

#### 方案 1：使用已知可用的基础
1. 回滚到 Build #33 的状态
2. 在此基础上逐步添加修复
3. 每次修改后验证 Android 目录结构

#### 方案 2：完全手动创建 Android 项目
1. 不依赖 `flutter create`
2. 手动创建所有必要的 Gradle 文件
3. 确保文件结构和 Build #33 一致

#### 方案 3：使用 Flutter 官方推荐方式
1. 本地运行 `flutter build apk` 验证
2. 将本地成功的配置复制到 CI
3. 确保 CI 环境和本地一致

## 建议的下一步

### 立即停止
- 停止创建新的工作流文件
- 停止在 CI 中试错

### 立即执行
1. **本地验证**：在本地运行 `flutter build apk`，确保能成功
2. **对比差异**：比较本地和 CI 的 Android 目录结构
3. **固定配置**：将本地成功的配置提交，不再依赖 `flutter create`

### 长期改进
1. 建立 CI 前的本地验证流程
2. 使用 Docker 确保环境一致性
3. 建立构建失败的快速回滚机制

## 教训

1. **不要重复同样的修复**：如果同样的错误发生超过 3 次，说明修复策略有问题
2. **简化而不是复杂化**：当问题复杂时，应该简化而不是添加更多逻辑
3. **验证再提交**：每次修改应该在本地或测试环境验证
4. **理解工具行为**：在修复前应该理解 Flutter/Gradle 的实际行为
