### 2.2 技术实现要点

#### API 接口规范确认

```yaml
# 反馈提交接口
POST /api/feedback/submit
Request:
  type: string          # 问题类型
  module: string        # 功能模块
  severity: string      # 严重程度
  description: string   # 问题描述（10-500字）
  images: array         # 图片URL数组（最多3张）
  contact: string       # 联系方式
  attach_log: boolean   # 是否附加日志
  device_info: object   # 设备信息（自动）
  context: object       # 上下文信息（自动）

Response:
  feedback_id: string   # 反馈ID
  status: string        # pending
  created_at: timestamp
```

#### 数据库表结构确认

```sql
CREATE TABLE feedbacks (
    id SERIAL PRIMARY KEY,
    user_id VARCHAR(64),
    type VARCHAR(32) NOT NULL,
    module VARCHAR(32) NOT NULL,
    severity VARCHAR(16) DEFAULT 'medium',
    description TEXT NOT NULL,
    images JSON,
    contact VARCHAR(64),
    device_info JSON,
    context JSON,
    status VARCHAR(32) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

## 3. 反馈入口功能测试计划

### 3.1 测试范围

| 测试项 | 测试内容 | 预期结果 |
|--------|----------|----------|
| 入口可见性 | 设置页"帮助与反馈"按钮 | 按钮可见，点击跳转 |
| 表单完整性 | 类型/模块/描述必填验证 | 空值提示，正确拦截 |
| 图片上传 | 选择/拍照/压缩/预览 | 最多3张，单张<5MB |
| 网络异常 | 弱网/断网提交 | 本地缓存，恢复后重试 |
| 成功反馈 | 完整提交流程 | 成功提示，返回列表 |

### 3.2 测试用例

```yaml
TestCase 1: 正常反馈提交
  前置: 用户已登录，网络正常
  步骤:
    1. 进入设置页，点击"帮助与反馈"
    2. 选择问题类型"功能异常"
    3. 选择功能模块"导航"
    4. 输入描述"导航语音不播报"
    5. 上传1张截图
    6. 填写联系方式
    7. 点击提交
  预期: 提交成功，显示反馈ID，后台可见

TestCase 2: 必填项验证
  前置: 进入反馈页面
  步骤:
    1. 直接点击提交按钮
  预期: 提示"请选择问题类型"，提交被拦截

TestCase 3: 网络异常处理
  前置: 进入反馈页面，填写完整信息
  步骤:
    1. 断开网络
    2. 点击提交
    3. 恢复网络
  预期: 提示网络异常，恢复后自动重试提交

TestCase 4: 图片数量限制
  前置: 进入反馈页面
  步骤:
    1. 尝试添加第4张图片
  预期: 提示"最多上传3张图片"，无法添加
```

## 4. 收集流程验证

### 4.1 完整流程图

```
用户提交反馈
    ↓
[客户端] 表单验证
    ↓
[客户端] 图片上传（如有）
    ↓
[客户端] 组装数据（含自动信息）
    ↓
[API] 接收请求，数据验证
    ↓
[API] 写入数据库
    ↓
[API] 发送飞书通知（P0问题）
    ↓
[后台] 反馈进入待处理列表
    ↓
[产品/运营] 人工审核
    ↓
    ├─→ 重复问题 → 关联已有工单
    ├─→ 已修复 → 回复用户
    ├─→ 需要更多信息 → 联系用户
    └─→ 确认问题 → 创建Bug工单
```

### 4.2 SLA 验证清单

| 严重程度 | 首次响应时效 | 验证方法 |
|----------|--------------|----------|
| critical | 2小时 | 模拟P0问题，计时通知到达时间 |
| high | 24小时 | 检查飞书通知是否发送 |
| medium | 72小时 | 检查后台列表是否正常显示 |
| low | 7天 | 定期统计报告 |

### 4.3 数据质量检查点

| 检查项 | 检查方法 | 合格标准 |
|--------|----------|----------|
| 设备信息完整性 | 抽查10条反馈 | 100%包含平台/版本/设备 |
| 上下文准确性 | 抽查导航相关反馈 | 位置/路线ID与描述匹配 |
| 日志有效性 | 抽查含日志的反馈 | 日志可读，包含错误信息 |
| 图片可访问性 | 抽查含图片的反馈 | 图片链接有效，内容相关 |
